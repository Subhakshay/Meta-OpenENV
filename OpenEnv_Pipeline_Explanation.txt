=============================================================================
CUSTOMER SUPPORT TRIAGE: OPENENV PIPELINE & ARCHITECTURE EXPLANATION
=============================================================================

1. THE BIG PICTURE: WHAT IS THIS PROJECT?
-----------------------------------------------------------------------------
This project is an AI evaluation environment compliant with the "OpenEnv" standard. 

Instead of testing an AI model (like Groq's LLaMA-3) on static multiple-choice questions, this project drops the AI into a simulated, interactive "game": a Customer Support Helpdesk. 

The AI plays the role of a customer support agent. It reads a queue of incoming support tickets and must decide how to classify them (Priority and Category), what to reply to the customer, and whether to escalate the issue to human managers. Crucially, the AI's actions have real, rippling consequences on the simulated company's bottom line.


2. THE ENGINE: PROCEDURAL TICKET GENERATION
-----------------------------------------------------------------------------
If the AI sees the same tickets every time, it can simply memorize the answers. To prevent this, the environment uses "Procedural Generation" in `environment.py`.

- Blueprints: There are templates (blueprints) for different types of tickets. For example, a "Critical Server Outage" template might look like: "Our {tier} integration has been down for {duration}. We are losing ${loss} per minute."
- Variable Pools: There is a dictionary holding hundreds of combinations for these placeholders (e.g., {duration} could be "2 hours" or "since yesterday").
- Generation: When a session starts, the engine picks blueprints, picks variables randomly, and stitches them together. The result is millions of unique tickets. The AI is forced to actually read and understand the text.


3. THE THREE TASKS (DIFFICULTY SCALING)
-----------------------------------------------------------------------------
The environment challenges the AI across three tasks of increasing difficulty:

- Task 1 (Easy): The AI only has to guess the "Priority" (Low, Medium, High, Critical) of very obvious tickets.
- Task 2 (Medium): The AI has to guess the "Priority", the "Category" (Billing, Technical, Shipping, etc.), and draft a professional text response. Some tickets are slightly ambiguous.
- Task 3 (Hard, Multi-Turn): The AI is faced with highly ambiguous tickets. The AI must realize it does not have enough information and take an action to "ASK" the customer a clarifying question. The customer (simulated by the script) replies, and only then can the AI correctly classify it.


4. THE STATEFUL WORLD (BUSINESS CONSEQUENCES)
-----------------------------------------------------------------------------
The environment tracks a "WorldState". The AI isn't just grading tests; its choices mutate the world:
- Company Balance: Starts at $10,000. If the AI wrongfully escalates a billing issue, the system processes a refund, lowering the balance.
- Churn Risk: Starts at 0%. If the AI drafts a terrible response, the risk of the customer leaving spikes by 15%.
- Escalation Queue & SLA Breaches: If the AI escalates too many things, the queue overflows and penalizes the score. If the AI misses a CRITICAL ticket, an SLA Breach is triggered.


5. THE OPENENV API: HOW IT INTERACTS
-----------------------------------------------------------------------------
Because this is an OpenEnv standard project, it wraps the entire simulation in a FastAPI web server (`main.py`). The outside world (or the AI) interacts with the simulation via HTTP endpoints. 

A standard lifecycle looks like this:

Step A: POST /reset
- The client tells the server, "Start a new game on Task 2".
- The server generates 10 fresh, procedural tickets.
- It returns a unique `session_id` and the text of the very first ticket as the "Observation".

Step B: POST /step (The Loop)
- The client AI reads the ticket, thinks, and sends its action back to the server: {"assign_priority": "high", "assign_category": "technical", "escalate": false}.
- The Python server receives this and mathematically grades it (Reward Calculation). 
- If the AI guessed correctly, it gets a high reward. The server also mutates the WorldState based on the action.
- The server returns the mathematical reward, the mutated WorldState, and the NEXT ticket in the queue.
- This repeats until the queue is empty (where a flag `done=True` is triggered).


6. THE EVALUATOR: HOW THE AI IS GRADED (NO LLM USED)
-----------------------------------------------------------------------------
A common misconception is that another AI is grading the submissions. This is false. The grading system is 100% deterministic Python code.

- Priority/Category Grading: When the ticket was procedurally generated, the engine secretly stored the "True Priority". The Python code simply checks if the AI's answer == True Priority. 
- Response Grading: The Python code uses a checklist. Did the AI say "Hello"? Did it reuse keywords from the ticket subject? Did it use words like "investigate" or "fix"? Points are added mathematically based on strict string matching.


7. THE DATABASE LAYER (`db.py`)
-----------------------------------------------------------------------------
As the API processes `/reset` and `/step`, it silently records everything to a database (SQLite or Postgres) via `db.py`.

- `episodes` table: When `/reset` is called, a new row is created. When the 10th step finishes (`done=True`), the server calculates the final mean reward, final company balance, and final SLA breaches, and updates this row to close it.
- `steps` table: Every time `/step` is called, a row is inserted logging the exact action the AI took and the exact fractional reward it earned.
- `world_state_snapshots` table: Every step, it records the exact company balance and churn risk, creating a timeline of the simulation.
- `tickets_log` table: Every ticket generated is stored here so human researchers can review them later.

The endpoint `GET /episodes` simply queries the database for all rows in the `episodes` table where `closed_at` is not empty, returning the final dashboard metrics.


8. LOCAL INFERENCE vs SERVER API MODE
-----------------------------------------------------------------------------
There are two ways to run the environment, which often causes confusion:

Mode 1: The API + Test Script (`test_db.py`)
This is running the actual web server. The Python script sends HTTP requests over the network layer to `localhost:7860`. Because it hits the API endpoints, the API triggers `db.py` to save the records to the database.

Mode 2: Local Python Execution (`inference.py`)
This bypasses the web server entirely! The script imports the `CustomerSupportEnv` object natively into Python. The Groq LLM talks directly to the Python object. Because it skips the FastAPI network layer (`main.py`), it completely skips all database logging capabilities. This is purely for fast, raw testing in the terminal.
