# The Gauntlet + Shifting Sands — Implementation Summary

In this session, we successfully built Phase 2 of the OpenEnv Customer Support Triage environment. The goal was to transition from a static, rules-based simulation to a dynamic, self-play adversarial environment with shifting policies. 

The implementation resides primarily in the new `gauntlet_shifting_sands` package. Below is a breakdown of the core components developed:

## 1. Policy Engine (`policy.py`)
- Created the `PolicyRegistry` and `PolicyVersion` dataclasses.
- Defined three distinct policy configurations (v1, v2, v3) that govern refund windows, SLA thresholds, valid ticket categories, required response greetings, and ticket schema fields.

## 2. Stateful World & Curriculum (`world_state.py`)
- Implemented the `WorldState` dataclass to persist state within an episode (e.g., `company_balance`, `churn_risk`, `escalation_queue_size`, `sla_breaches`).
- Added a cross-episode curriculum controller: It tracks the `attacker_win_rate_50` across episodes and dynamically adjusts the `difficulty_level` up or down based on the Defender agent's performance.

## 3. Deterministic Reward System (`rewards.py`)
- `calculate_defender_reward`: Built a multi-component scoring function covering priority accuracy, category accuracy, response quality (via regex heuristics), correct escalation, and refund decisions.
- Added strict checks for "Stale Decisions" (using old policies post-drift) and a robust "Hallucination Checker" that scans agent text for policy claims (e.g., SLA hours, refund days) that do not exist in *any* policy version.
- `calculate_attacker_reward`: Developed an inverse reward function where the Attacker earns points by causing the Defender to misclassify tickets or make incorrect refund decisions.

## 4. Policy Drift Injection (`drift_scheduler.py`)
- Developed the `DriftScheduler` to inject sudden policy changes during an episode.
- Pre-configured two events:
  - Step 4 (v1 → v2): Tightens SLA windows and refund eligibility, and mandates professional greetings.
  - Step 9 (v2 → v3): Introduces a new "Security" category and expands the observation schema to include `sentiment_score` and `account_age_days`.

## 5. Core Environment & Ticket Engine (`environment.py`)
- Implemented the `CustomerSupportEnv` providing the Gym-style `reset()` and `step()` API.
- Designed 30+ ticket blueprints and variable pools (e.g., tiers, symptoms, durations, products) for procedural generation.
- Added support for three task levels: Priority Only (easy), Full Classification (medium), and Multi-Turn (hard), including a simulated customer reply function for clarification loops.

## 6. Adversarial Attacker Agent (`attacker.py`)
- Built the `AttackerAgent` which generates batches of deceptive tickets at the start of an episode.
- Implemented 6 distinct deception strategies: Priority Camouflage, Fake Urgency, Category Confusion, Boundary Exploitation, Emotional Manipulation, and Schema Exploitation.
- Uses LLM calls (via Groq/Llama-3) to generate tickets based on strategy prompts, with a procedural fallback mechanism to ensure environment stability.

## 7. API and Persistence (`main.py` & `db.py`)
- `main.py`: Set up the FastAPI server complying with OpenEnv specifications (`/reset`, `/step`, `/episodes`, `/world_state`, `/health`). Manages session states and invokes the Attacker agent on `/reset`.
- `db.py`: Created an asynchronous SQLite database layer (using `aiosqlite`) to persist `episodes`, `steps`, `world_state_snapshots`, `tickets_log`, and `drift_events` for extensive analytics.

## 8. Testing and Manifest (`inference.py`, `openenv.yaml`, `requirements.txt`)
- Provided `inference.py` as a local test runner that bypasses the FastAPI server. It includes both a heuristic rule-based agent and an LLM-based defender agent.
- Updated the `openenv.yaml` manifest to reflect the new framework, action/observation spaces, and tasks.
- Verified functionality by running the inference test script inside the virtual environment. All tasks completed successfully, confirming proper integration of the attacker, drift mechanisms, and reward calculations.
